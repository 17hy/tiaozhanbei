## 1. 执行摘要

基于当前工作区已落地的训练与部署链路，最适合继续深挖的方向不是切换到 Transformer 检测器，而是围绕 `YOLOv8-s-P2 + RK3588 INT8` 做数据、训练、量化和部署四个层面的协同优化。

核心原因有三点：

1. 本地验证集 `8700` 张图、`69579` 个框中，约 `74.08%` 的目标框面积低于图像面积的 `1%`，属于典型的小目标主导数据集。
2. 当前仓库已经具备 `RK3588 + RKNN + INT8 + zero-copy` 的完整工程基础，继续沿用 CNN 检测器的投入产出比最高。
3. 从公开论文和官方实现看，RT-DETR/RT-DETRv2、YOLOv10 等方法在通用 COCO 上很强，但在 `RK3588` 这种强约束边缘平台上，部署复杂度、算子兼容性和端到端收益未必优于已经工程化完成的 `YOLOv8-s-P2`。

结论：

- 如果目标是 `1080P 实时边缘推理`，推荐主线方案为 `YOLOv8-s-P2 -> INT8 量化 -> zero-copy 部署 -> 小目标增强训练`。
- 如果目标是追求最高精度而不强卡 `23ms` 延迟，可增加 `切片推理/切片微调（SAHI）` 作为高精度备选。
- 如果后续需要论文式对比实验，建议把 `YOLOv8-s`、`YOLOv8-s-P2`、`YOLOv10-s`、`RT-DETR-R18/RT-DETRv2-R18` 作为四组主比较对象。

## 3. 当前仓库的工程基础

当前工作区已经具备以下关键能力：

1. `RKNN` 转换脚本已支持 `rk3588` 和 `INT8`：`training_files/pt2rknn.py`
2. `RKNN` C++ 推理链路已存在普通版和 `zero-copy` 版：`training_files/yolov8/cpp/rknpu2/yolov8.cc`、`training_files/yolov8/cpp/rknpu2/yolov8_zero_copy.cc`
3. 输入预处理已通过 `uint8 + NHWC` 方式与 NPU 前处理融合，减少 CPU 侧开销。
4. 本次已补齐 `YOLOv8s-P2` 模型配置：`training_files/model_configs/yolov8s-p2.yaml`
5. 本次已将后处理从固定 `3` 头扩展为自动识别 `3/4` 头，能够正确支持 `P2-P5` 检测头输出。

这意味着：

- 从“能不能做”看，`YOLOv8-s-P2 + RK3588` 已经具备工程闭环。
- 从“值不值得换模型”看，现阶段优先继续优化现有链路，比贸然切到全新检测器更稳。

## 4. SOTA 路线对比

### 4.1 对比对象

结合当前数据分布、工程基础和边缘部署目标，本报告建议重点比较以下四条路线：

| 路线 | 特点 | 对 RK3588 适配性 | 结论 |
|---|---|---|---|
| YOLOv8-s | 当前基线 | 高 | 必须保留作基线 |
| YOLOv8-s-P2 | 增强小目标能力 | 很高 | 推荐主线 |
| YOLOv10-s | 更强效率设计，NMS-free | 中 | 可做候选，但需验证 RKNN 兼容性 |
| RT-DETR-R18 / RT-DETRv2-R18 | 端到端检测，精度强 | 中低 | 更适合论文对比，不宜直接当主部署线 |

### 4.2 SOTA 结论

#### 方案 A：YOLOv8-s-P2

优势：

- 与本地数据的小目标分布高度匹配。
- 仍属于 CNN/FPN 路线，导出 ONNX、转 RKNN、做 INT8 量化都更稳。
- 与现有部署代码兼容成本最低。

风险：

- 比标准 `YOLOv8-s` 多一个检测头，后处理和中间特征图开销会增加。
- 如果训练仍使用 `640` 且无重平衡增强，`P2` 的提升会被数据偏置部分抵消。

结论：

- 在当前项目约束下，这是“精度-延迟-可部署性”综合最优方案。

#### 方案 B：YOLOv10-s

公开论文显示，YOLOv10 通过一致性双分配与整体式效率设计，把 YOLO 系列进一步推向 `end-to-end` 与更高效率边界；论文中给出 `YOLOv10-S` 在相近 COCO AP 下比 `RT-DETR-R18` 更快，且参数与 FLOPs 更低。

优势：

- 在通用实时检测上有很强的性价比。
- 理论上后处理更轻，端到端延迟有优势。

风险：

- `RKNN` 对新算子图和端到端检测头的兼容性需要单独验证。
- 即使 PyTorch/ONNX 侧可跑，Rockchip NPU 上未必能完整保留其 NMS-free 设计收益。

结论：

- 适合作为“更先进 YOLO 架构”的候选对比项。
- 不建议在没有兼容性验证前替代当前主线。

#### 方案 C：RT-DETR / RT-DETRv2

公开论文显示，RT-DETR 是主打实时端到端检测的 Transformer 路线；RT-DETRv2 进一步加入 selective multi-scale feature extraction、离散采样替代和训练策略优化，提升灵活性和可部署性。

优势：

- 在 COCO 等通用基准上具备很强的精度竞争力。
- 端到端检测天然减少传统 NMS 的一部分后处理约束。

风险：

- 对边缘 NPU 的算子支持和图优化要求更高。
- Transformer 解码器通常对算力、访存和部署框架更敏感。
- 在 `RK3588` 这种强实时、强工程约束平台上，未必能兑现论文中的速度优势。

结论：

- 更适合作为“论文型对比模型”。
- 如果目标是尽快做出稳定可交付系统，不建议作为第一优先级。

#### 方案 D：YOLOv8-s-P2 + SAHI

SAHI 是面向小目标的切片推理/切片微调框架，官方文档给出其在小目标场景中可显著提升 AP。

优势：

- 对极小目标和高分辨率大图特别有效。
- 适合 1080P/4K 图像中远距离目标极小的场景。

风险：

- 切片推理会显著增加延迟。
- 对“单路实时 23ms”目标不友好。

结论：

- 适合作为“高精度离线方案”或“报警复核方案”。
- 不适合作为当前实时主线。

## 5. 为什么当前最优解仍然是 YOLOv8-s-P2

这是基于本地数据与公开文献的综合推断。

### 5.1 与数据分布最匹配

本地验证集里：

- `74.08%` 目标面积低于 `1%`
- `57.44%` 目标面积低于 `0.5%`
- 长尾类别严重，尾类本就更容易在小目标场景下被漏检

因此：

- 增加 `P2` 分支能更早保留高分辨率空间细节。
- 相比盲目换成更重的 backbone，`P2` 更直接命中当前误差来源。

### 5.2 与 RK3588 的部署约束最匹配

当前项目已经有：

- `INT8` 量化链路
- `zero-copy` 推理实现
- `RKNN` 输入输出内存绑定
- `P2` 后处理适配

因此主线最优策略不是“大换血”，而是“把现有链路做满”。

### 5.3 性价比高于 RT-DETR 路线

从论文看，RT-DETR/RT-DETRv2 在通用 benchmark 上很强，这一点没有问题。  
但对于本项目，真正的目标函数不是“单纯 AP 最大化”，而是：

`边缘设备可部署性 + 小目标召回 + 1080P 实时性 + 工程可维护性`

在这个目标函数下，`YOLOv8-s-P2` 更优。

## 6. 优化建议与优先级

以下建议按“投入产出比”排序。

### 6.1 第一优先级：继续强化 P2 主线

建议：

1. 保持 `YOLOv8-s-P2` 作为主结构。
2. 基线对比至少保留：
   - `YOLOv8-s`
   - `YOLOv8-s-P2`
3. 训练侧优先观察 `AP_small`、`Recall_small`、尾类召回率，而不只看总 mAP。

预期收益：

- 对小目标召回最直接。
- 工程改动最小。

### 6.2 第二优先级：做类别重平衡训练

原因：

- 前三类占比 `76.37%`，尾类严重不足。

建议：

1. 对尾类样本做 oversampling。
2. 对尾类使用 copy-paste 或 targeted augmentation。
3. 在训练集层面做 class-aware sampling。
4. 若业务允许，可将尾类单独做 hard example mining。

预期收益：

- 尾类召回提升通常会明显高于总 mAP 提升。

### 6.3 第三优先级：提高训练分辨率，但部署分辨率保持可控

建议：

1. 训练做 `640 -> 800/960` 的两阶段训练。
2. 部署默认仍以 `640` 输入为主，先保实时性。
3. 对关键场景再尝试 `800` 输入的性能上限。

原因：

- 当前数据中存在大量 `1080P/4K` 图像，小目标经缩放后信息损失大。
- 提高训练分辨率对小目标有帮助，但部署端必须平衡延迟。

预期收益：

- 对极小目标精度通常有正收益。
- 延迟会变差，需要单独做板端 Profiling。

### 6.4 第四优先级：优化 INT8 校准集

当前仓库的 `pt2rknn.py` 已支持 `INT8`。

建议：

1. 校准集不要随机抽样，优先覆盖：
   - 极小目标
   - 密集场景
   - 尾类样本
   - 强背光/低照度场景
2. 控制校准集分布，使其更接近真实业务流。
3. 比较 `FP` 与 `INT8` 的 `AP_small` 差值，重点看小目标是否被量化伤害。

原因：

- 对小目标检测来说，INT8 量化误差更容易体现在低对比、小纹理目标上。

### 6.5 第五优先级：部署端继续压缩 CPU 开销

当前代码中已经具备较好的工程基础，建议继续做以下动作：

1. 优先使用 `zero-copy` 路径。
2. 保持 `uint8` 输入，让预处理尽可能并入 NPU 输入链路。
3. 降低不必要的 CPU 后处理开销，避免重复内存拷贝。
4. 对后处理阈值做业务化调优，而不是只用固定 `0.25/0.45`。

原因：

- 对 RK3588 这类边缘平台，端到端延迟不只取决于模型本身，还强依赖内存搬运和 CPU 后处理。

### 6.6 第六优先级：把 SAHI 作为高精度备选，而不是实时主线

建议：

1. 对离线批处理、重点截图复核、报警二次确认场景使用 SAHI。
2. 不把 SAHI 直接放进默认实时链路。

原因：

- 它更适合“精度优先”，不适合“时延优先”。

## 8. 最终建议

### 8.1 主线建议

主线采用：

`YOLOv8-s-P2 + 类别重平衡训练 + 高质量 INT8 校准 + RK3588 zero-copy 部署`

这是当前最符合“数据分布 + 工程现状 + 边缘实时”三重约束的方案。

### 8.2 备线建议

备线分为两条：

1. `YOLOv10-s`：验证是否存在更高的速度/精度上界。
2. `YOLOv8-s-P2 + SAHI`：验证高精度离线方案上限。

### 8.3 不建议的方向

在没有额外 GPU 训练资源和 RKNN 兼容性验证前，不建议把 `RT-DETR/RT-DETRv2` 直接作为主部署路线。  
原因不是它不强，而是它对本项目目标函数并不占优。

## 9. 结论

本地数据已经给出非常明确的信号：

- 数据是小目标主导。
- 类别分布严重长尾。
- 图像分辨率混杂，高分辨率源图较多。

因此，最合理的路线不是单纯追逐“最新模型名称”，而是把模型结构、数据分布、量化策略和板端部署统一起来。  
对于当前项目，`YOLOv8-s-P2` 就是这个统一点。

如果只保留一句话结论：

`这套数据更需要 P2、小目标重平衡和量化校准，而不是优先切换到更复杂的检测器。`
